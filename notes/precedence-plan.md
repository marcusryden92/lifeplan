## Context

Goals conflate "completable outcome" with "ordered container". Work streams (e.g. a Work pipeline) need ordering without finitude; root-level items need prerequisite links ("get married" needs "find a girlfriend" AND "get an apartment") that no container expresses. Two concepts, deliberately distinct:

- **Queue (pipe)** — a first-class entity (NOT a plannerType): a persistent, user-authored, ordered list of references to existing root-level Planner items (tasks and goals). No endpoint; members exit in the order they entered. Managed at `/queues`.
- **Dependency** — a logical prerequisite edge between root-level items. Multi-predecessor, order among prerequisites irrelevant, authored in the item editor. No container entity — the connected component of the dependency graph IS the "project"; it is derived, never declared. No Project model, ever.

Both compile to one engine representation: `PrecedenceEdge { fromId, toId, source: "queue" | "dependency", queueId? }`. The engine gates each candidate on max end across all placed predecessors and never sees a cycle — legality is enforced at authoring time.

### Decided semantics (these govern all current and future surfaces)

1. **All members/successors schedule (full forecast)**, each bounded via the engine's existing `afterTime` seam — `findValidSlots` owns `effectiveAfter = max(afterTime, earliestStartDate chain)`, so precedence composes with per-item constraints for free. Never re-implement the max at a gate.
2. **Edge scope.** Both endpoints root-level, triaged, `task | goal`. Plans excluded (deferred). A planner belongs to at most one queue (DB unique on `QueueMember.plannerId`); dependencies make the merged graph a general DAG.
3. **Orthogonality.** Pipe membership and dependency are independent. A pipe stalls at a member with an unmet external dependency; successors never flow around a stalled member.
4. **Merged-graph validation, logical order.** Cycle checks run against dependency edges + each queue's FULL logical member order (completed/unready included — transparency affects gating, never legality). Validated on every mutation of either system. The engine never arbitrates.
5. **Redundant same-pipe dependencies are allowed**; contradictions are blocked with the cycle path shown.
6. **Transparency (engine gating).** Completed members/predecessors: transparent, no bound (both sources). Unready-goal predecessor: queue → silent chain-through; dependency → successor schedules unbounded + LOUD message. Asymmetry intentional.
7. **Permanent placement failure** of a predecessor → successors schedule anyway, unbounded, plus a message. Pure budget exhaustion emits a distinct "past the horizon" flavor (`SEQUENCE_PAST_HORIZON`) rather than pretending the sequence broke.
8. **Readiness gate (authoring-time).** A goal cannot be readied while any dependency predecessor is an unready goal ("Awaiting <title>", with per-blocker Ready/Disconnect shortcuts, no deep cascade); un-readying a goal a READY goal depends on is refused symmetrically. The engine keeps the defensive loud fallback (#6) for states that slip through.
9. **Queue's optional `categoryId` = inherited default** for members with no effective category of their own. Dependencies carry no category semantics.
10. **Future graph surfaces must require zero new rules** — they read the same data and write through the same validators/choke points.

## 2. Deviations from the original plan / gotchas

- **The keyed-object refactor of the sync seam HAPPENED** (the plan's "preferred" option): `compareData`/`handleServerTransaction` now take a single `SyncInputs` object of `{current, previous}` groups instead of ~20 positional params. Any future sync group extends `SyncInputs` + `DatabaseChanges` + `runSync`'s literal.
- **Watermark capacity exclusion (new engine behavior, load-bearing).** `scheduleTasksAndGoals` now also excludes from watermark sizing any candidate whose `effectiveCandidateDuration` exceeds `maxEffectiveCapacityFor` (horizon-independent weekly ceiling). Without it, a TOO_LARGE candidate pinned `biggestFit < biggestRemaining` and burned all 12 expansions before the walk could fail it — which misclassified precedence breaks as `"horizon"` instead of `"failed"` (and was a pre-existing waste for plain too-large tasks). Covered by queue-sequence test 5.
- **Dependency picker degrades "disabled with reason" to "annotated + hard-check"**: `Combobox` has no per-option disable, so cycle-blocked candidates are listed with "— would create a loop" and the commit refuses with the full `describeCycle` path. The one-click fix offer (semantics #5: "move D ahead of B") is NOT shipped — fast-follow as planned.
- **DependencyGatePopover uses Radix Popover** (the Combobox precedent), not `usePopoverPosition`.
- **Test-pattern split**: queue-sequence + dependency-gate use the live-data fixture as planned; precedence-constraints-compose + queue-category-inheritance are hand-built minimal geometry (split-task/goal-day-cap precedent) because their assertions need controlled slot fabric.
- **Identity discipline is everywhere and tested** — prune helpers, `applyQueueCategoryInheritance`, `mutateQueueMembers.removeQueueMember` all return the same reference on no-op. Keep it that way; phantom diffs make second-window syncs permanently stale.
- Queue rows created client-side carry the provider `userId`; member rows are minted in the choke point with fresh uuid + timestamps. Queue edits do NOT restamp `updatedAt` client-side (categories precedent); member reorders do (harmless, value-based diff).

### 4d. Goal linked into another goal as a detour (TODO's "goal as subtask") — IMPLEMENTED 2026-07-13

**Shipped via the placeholder model (see the STATUS note at the top), not the columns-on-the-linked-goal design sketched below.** The analysis below is preserved for context; the shipped semantics live in CLAUDE.md.

NOT reparenting and NOT containment. The linked goal stays a separate root — own category, own constraints, own readiness, own item page — and a link marks a position inside the host goal's tree. Scheduling is a **traversal-time splice**: the host's leaf enumeration inlines the linked goal's sorted bottom layer at the link position, and the existing `goalAfterTime` chaining does the rest. Zero new gate mechanics; the edit lives where `scheduleGoal` enumerates leaves (`getSortedTreeBottomLayer` consumer), not in the precedence gate.

- Falls out free: `buildPlannerCategoryMap` / `buildPlannerConstraintsMap` walk the real `parentId` chain, so the linked goal's leaves keep ITS category and constraints while riding the host's pacing; the root stays root, so central pruning never bites.
- Data model (decided): NOT redirect pointers on neighboring tasks (the retired `dependency` linked-list column reborn — see `drop_planner_dependency`), and NOT a placeholder child row (pseudo-plannerType leaks into every planner consumer). One fact on the LINKED goal: two nullable Planner columns — `detourHostId String?` (FK → Planner, `onDelete: SetNull`) + `detourSortOrder Float?` (position on the host parent's fractional sibling axis; inert once `detourHostId` is null). The scalar IS the one-link invariant; "is linked" = `detourHostId != null`; the columns ride the existing planner sync/diff/spread-preserve machinery with zero new plumbing (the `earliestStartDate`/`allowedTimes` reasoning). Root-goals-only, stale values on retyped/nested rows inert (`maxMinutesPerDay` pattern). Real cost: sibling enumeration merges two sources on one fractional axis (own children by `sortOrder` + incoming detours by `detourSortOrder`) — build the detours-by-host index in the memoized per-array pass `getSortedTreeBottomLayer` already runs. Central pruning gains `pruneDetours`: clear the pair when the host node no longer resolves into a triaged root goal's tree (retype/untriage; deletes covered by SetNull).
- Decisions the implementation owes: (1) a linked goal stops being an independent candidate while linked (else its leaves place twice) — constrains the SPLICE only, never prerequisite fan-in/out ("needed by many things" stays dependency edges); queue membership simplest as mutually exclusive with being linked; (2) day caps compose pointwise-min across host and linked goal (the split × goal-cap budget seam is the template); (3) transparency: completed linked goal → chain passes through; unready → pick skip-silently (queue semantics) vs stall-loud (dependency semantics); (4) a linked goal with its own dependency predecessors makes the host walk consult the gate MID-walk — v1 can reject links on goals with incoming edges; (5) validation: contract host + linked goal into ONE node (transitively across link chains) in the merged graph — this is why `PrecedenceEdge.source` and the validation builder stayed extensible rather than a closed union.
- For plain "goal X needs goal Y" with no interleaving, the shipped dependency edge is still the answer — the detour is for when Y's work belongs INSIDE X's sequence.

### 4e. Subtree-level dependencies — decided against (2026-07-13), possibly permanently

Root-only endpoints keep dependencies a top-level concept (matching root-only `categoryId`/`color`/queue-membership) and the authoring surface holdable-in-head. Preserved analysis, split by which SIDE goes interior:

- Interior SUCCESSORS only (mid-chain leaf gating on external roots) are **position-independent**: attach-time validation suffices, subtask reorders stay free; engine cost is the same mid-walk gate consult the detour's decision (4) needs. Safe to add later without taxing any drag surface.
- Interior PREDECESSORS are **position-dependent**: every goal's internal leaf order joins the merged cycle graph, so EVERY subtask drag becomes a cycle-validated mutation across the library tree, subtasks page, and assistant move ops. That reorder tax plus conceptual load is the reason for the skip.
- Workarounds inside root-only scope: depend on the whole goal (coarser), or promote the prerequisite subtask to its own root and depend on that (composes with a detour re-splicing it into its goal's chain).
